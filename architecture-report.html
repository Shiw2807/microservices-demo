Critical Risks

CRITICAL All gRPC Communications are Plaintext — Payment Card Data Exposed on Wire
Every inter-service gRPC call uses grpc.WithInsecure() (clients) and grpc.ServerCredentials.createInsecure() (paymentservice server). Credit card data traverses the full path — browser → frontend → checkoutservice → paymentservice — entirely unencrypted. Any pod with cluster network access can perform a man-in-the-middle attack and capture payment data.
Worst case: Attacker deploys a rogue pod in the namespace, intercepts gRPC streams, captures all credit card numbers, CVVs, and billing addresses in transit.
src/frontend/main.go:mustConnGRPC (grpc.WithInsecure) · src/paymentservice/server.js:createInsecure()
CRITICAL No Service-to-Service Authentication — PaymentService Open to Any Pod
PaymentService (and all internal services) have zero authentication. No mTLS, no service accounts, no API keys. NetworkPolicies are opt-in only (authorizationPolicies.create: false in Helm). Any pod in the Kubernetes cluster — including a compromised workload — can call PaymentService.Charge with arbitrary parameters.
Worst case: Compromised pod calls PaymentService.Charge with crafted amounts and arbitrary card data, processing fraudulent transactions bypassing all business logic.
src/paymentservice/server.js · helm-chart/values.yaml (authorizationPolicies.create: false) · kubernetes-manifests/paymentservice.yaml

High Risks

HIGH Redis Cart Storage is Ephemeral (emptyDir) — All Cart Data Lost on Pod Restart
Redis uses an emptyDir volume. Any Redis pod restart, OOM eviction, or node migration destroys all shopping cart data permanently for every active user. Session cookies live 48 hours but the backing store can disappear at any time.
Worst case: Redis OOM-killed during peak traffic — all user carts wiped, checkout impossible until Redis restarts with empty state, no recovery path.
kubernetes-manifests/cartservice.yaml (volumes: - emptyDir: {})
HIGH Redis is a Single Point of Failure for All Checkout Operations
A single Redis pod with no replication, no sentinel, and no HA. cartservice is unavailable when Redis is down. checkoutservice calls cartservice for every order. Redis failure = complete checkout failure for all users simultaneously.
Worst case: Redis pod dies → cartservice throws → checkoutservice fails → all purchase attempts fail until Redis recovers.
kubernetes-manifests/cartservice.yaml · src/checkoutservice/main.go
HIGH Frontend Panics at Startup When SHOPPING_ASSISTANT_SERVICE_ADDR Missing
mustMapEnv is called unconditionally for SHOPPING_ASSISTANT_SERVICE_ADDR even when the shopping assistant feature is disabled by default. The env var is not set in the base kubernetes manifests. Any deployment that doesn't explicitly add this variable will CrashLoopBackOff immediately.
Worst case: Production upgrade without assistant component → frontend pod enters CrashLoopBackOff → entire storefront down.
src/frontend/main.go:mustMapEnv(&svc.shoppingAssistantSvcAddr, "SHOPPING_ASSISTANT_SERVICE_ADDR")
HIGH Session Cookie Missing Secure, HttpOnly, and SameSite Flags
The session cookie is set with only Name, Value, and MaxAge. Missing HttpOnly makes it readable by JavaScript (XSS). Missing Secure sends it over HTTP. Missing SameSite allows CSRF attacks on cart and checkout endpoints.
Worst case: XSS payload in ad content reads session cookie → attacker hijacks session → places fraudulent orders under victim identity.
src/frontend/middleware.go:http.SetCookie (no Secure/HttpOnly/SameSite)
HIGH No Rate Limiting on Frontend — DDoS Amplification Risk
The frontend HTTP server has no rate limiting middleware. With CPU limit of 200m and memory limit of 128Mi, the pod is trivially exhausted. currencyservice (highest QPS service) has no downstream rate protection either.
Worst case: Attacker floods frontend with requests → resource limits hit → frontend OOMs → all 11 microservices starved of requests, complete outage.
src/frontend/main.go · kubernetes-manifests/frontend.yaml (limits: cpu:200m memory:128Mi)
HIGH No Circuit Breakers in checkoutservice — Cascading Failure Risk
checkoutservice synchronously orchestrates 6 downstream services (cart, product catalog, shipping, payment, email, currency) with no circuit breakers, retries with backoff, or graceful degradation. A slow or unavailable dependency blocks the entire checkout flow.
Worst case: emailservice slow or rate-limited → all checkout goroutines block → checkout pod exhausts memory → complete checkout failure.
src/checkoutservice/main.go

Medium & Low Risks

MEDIUM gRPC Connection Timeout of 3 Seconds Causes CrashLoopBackOff on Cold Starts
All 8 gRPC connections must succeed within 3 seconds each or the frontend panics. Slow DNS, GKE node cold starts, or pod scheduling delays routinely exceed this. mustConnGRPC panics on timeout with no retry.
src/frontend/main.go:mustConnGRPC (context.WithTimeout time.Second*3)
MEDIUM No gRPC Server-Side Request Timeouts — Goroutine Leak Under Load
gRPC handlers in checkoutservice, shippingservice, and others have no deadline propagation or server-side timeouts. A stalled external call (e.g., payment processor timeout) holds a goroutine indefinitely.
src/checkoutservice/main.go · src/shippingservice/main.go
MEDIUM Hardcoded Shared Session UUID When ENABLE_SINGLE_SHARED_SESSION Enabled
When ENABLE_SINGLE_SHARED_SESSION=true, all users share the hardcoded session ID 12345678-1234-1234-1234-123456789123. Any user can view and modify any other user's shopping cart.
src/frontend/middleware.go:sessionID = "12345678-1234-1234-1234-123456789123"
MEDIUM cartservice Silent Fallback to In-Memory Store When No DB Configured
If REDIS_ADDR, SPANNER_PROJECT, SPANNER_CONNECTION_STRING, and ALLOYDB_PRIMARY_IP are all unset, cartservice silently uses an in-memory distributed cache. Cart data is per-pod and lost on every restart. No operator warning emitted.
src/cartservice/src/Startup.cs (AddDistributedMemoryCache fallback)
MEDIUM loadgenerator Deployed by Default in Production Helm Chart
loadGenerator.create: true in Helm values deploys a Locust load generator alongside production services by default, generating continuous synthetic shopping traffic that consumes real resources and inflates metrics.
helm-chart/values.yaml (loadGenerator.create: true)
