Executive Summary

Project: microservices-demo (Online Boutique) — 12-service e-commerce microservices application on Google Kubernetes Engine.

Overall SOC 2 Compliance Score: 34 / 100 — The application is not ready for a SOC 2 audit in its current state. Evidence collection across 7 Trust Services Criteria categories found 4 failing controls, 3 partial controls, and 0 fully passing categories.

Three critical gaps dominate the risk profile: (1) unpatched CVSS 9.8 arbitrary code execution in the payment service, (2) no authentication layer on any internal service endpoint, and (3) Kubernetes API tokens auto-mounted on every pod. These three gaps alone would block a SOC 2 audit.

On the positive side, the container runtime hardening is strong: all containers run non-root with capabilities dropped, read-only filesystems, and SHA256-pinned base images. GCP Secret Manager is used for database credentials. These controls provide a solid foundation to build on.

Estimated remediation timeline to audit-readiness: 6–8 weeks with dedicated engineering effort.

Act 2 — Gaps & Findings

Failing Controls

CC7.1 — Vulnerability Detection & Remediation Critical

File: src/paymentservice/package.json, src/cartservice/src/cartstore/AlloyDBCartStore.cs:69

The payment service carries a CVSS 9.8 arbitrary code execution vulnerability in protobufjs < 7.5.5 (GHSA-xq3m-2v4x-88gg) — unpatched and reachable at runtime. Five additional critical CVEs exist in the same transitive dependency tree. Separately, a SQL injection vulnerability in AlloyDBCartStore.cs:69 directly interpolates user-supplied userId and productId into raw SQL without parameterization. No dependency scanning, SAST, or container image scanning runs in the CI pipeline.

Evidence: npm audit: 5 critical, 6 high CVEs in paymentservice. protobufjs CVSS 9.8 (CWE-94). tar CVSS 8.2 path traversal. jws CVSS 7.5 HMAC bypass. SQL string interpolation confirmed at AlloyDBCartStore.cs:69.

No barrier   Attacker with network access to paymentservice can trigger RCE via malformed protobuf. SQL injection is exploitable by any authenticated frontend user.

Framework mapping: SOC2 CC7.1, CC7.3 | ISO 27001 A.12.6 | PCI-DSS 6.3.3

Fix: (1) Upgrade @opentelemetry/sdk-node to 0.217.0. (2) Downgrade @google-cloud/profiler to 0.1.4. (3) Replace @opentelemetry/exporter-otlp-grpc with HTTP exporter. (4) Parameterize all SQL in AlloyDBCartStore.cs using NpgsqlCommand with @param placeholders. (5) Add npm audit --audit-level=high and Trivy image scan to CI pipeline.
CC6.1 — Logical Access Controls & Authentication Critical

File: kubernetes-manifests/frontend.yaml:123 (all services)

No authentication or authorization is enforced on any service endpoint. All 12 microservices accept unauthenticated connections. The frontend is exposed externally over HTTP port 80 with no TLS. Inter-service gRPC calls carry no auth tokens. No API gateway or service mesh auth policy is configured in the base deployment.

Evidence: frontend-external Service type LoadBalancer, port 80 (frontend.yaml:130). No auth middleware found in any service source. Istio mTLS is opt-in overlay only (kustomize/components/service-mesh-istio/). No AuthorizationPolicies defined anywhere.

No barrier   Any pod inside the cluster can call any service without credentials. A compromised loadgenerator has a direct network path to paymentservice:50051.

Framework mapping: SOC2 CC6.1, CC6.6 | ISO 27001 A.9.1 | PCI-DSS 8.2

Fix: (1) Enable Istio service mesh with PeerAuthentication: STRICT mTLS. (2) Add AuthorizationPolicy per service. (3) Terminate TLS at the load balancer or ingress controller. (4) Enable authorizationPolicies.create: true in Helm chart values.
CC6.2 — Kubernetes API Token Auto-Mount Critical

File: All kubernetes-manifests/*.yaml ServiceAccount resources

Every pod in the cluster automatically mounts a Kubernetes ServiceAccount JWT token at /var/run/secrets/kubernetes.io/serviceaccount/token. No ServiceAccount sets automountServiceAccountToken: false. A compromised container can use this token to authenticate to the Kubernetes API server and enumerate Secrets, modify resources, or escalate privileges depending on cluster RBAC state.

Evidence: grep for automountServiceAccountToken returns 0 results across all manifests. redis-cart Deployment has no serviceAccountName — falls back to the default ServiceAccount.

Low complexity   Requires initial container compromise, then trivial API calls using the mounted token.

Framework mapping: SOC2 CC6.2, CC6.3 | CIS K8s 5.1.6 | ISO 27001 A.9.2

--- a/kubernetes-manifests/paymentservice.yaml
+++ b/kubernetes-manifests/paymentservice.yaml

   name: paymentservice
+automountServiceAccountToken: false
---

   serviceAccountName: paymentservice
+  automountServiceAccountToken: false
Fix: Add automountServiceAccountToken: false to all 12 ServiceAccount resources and all 12 Deployment pod specs. Add a dedicated ServiceAccount for redis-cart.

No barrier   Breaches go undetected. No forensic evidence available post-incident. Fails SOC 2 CC7.2 evidence requirement.

Framework mapping: SOC2 CC7.2, CC7.3 | ISO 27001 A.12.4 | PCI-DSS 10.2

Fix: (1) Promote google-cloud-operations overlay into base manifests. (2) Enable structured audit logging in checkout and payment flows. (3) Define log retention policy (minimum 90 days online, 1 year archived for SOC 2). (4) Set up alerting on error rate spikes and authentication anomalies.

Decisions to Make

Decision: Service Mesh vs Manual mTLS

Authentication between microservices is absent. Two paths exist: enable Istio (already partially configured as an optional overlay) or implement application-level auth tokens manually.

Cost of inaction: Any compromised pod has unauthenticated access to payment and checkout services. Fails SOC 2 CC6.1 indefinitely.
Option A: Enable Istio mTLS (Recommended)

Istio overlay already exists in kustomize/components/service-mesh-istio/. Set PeerAuthentication: STRICT + AuthorizationPolicies. ~1 week effort. Transparent to application code.

Option B: Application-level JWT auth

Add auth interceptors to each gRPC service. Higher effort (8–12 weeks), requires changes to all 12 services, higher maintenance burden. Full control over token lifecycle.

Recommendation: Option A. Istio infrastructure is already present and tested in CI. Enables mTLS and AuthorizationPolicies without application code changes.

Decision: Redis Persistence Strategy

Redis cart data uses emptyDir — every pod restart loses all active shopping carts. Two options for persistence.

Cost of inaction: Cart data loss on every deployment. Fails SOC 2 A1.2 availability requirement. Customer-facing data loss event.
Option A: Cloud Memorystore (Recommended)

Replace in-cluster Redis with GCP Memorystore. Managed backups, HA, no emptyDir. Memorystore kustomize component already exists at kustomize/components/memorystore/.

Option B: PersistentVolumeClaim

Add a PVC to the redis-cart Deployment. Cheaper, self-managed. Requires configuring backup jobs separately. Less HA than Memorystore.

Recommendation: Option A. Memorystore component already exists in the repo — lowest effort path to durable, HA cart storage.
